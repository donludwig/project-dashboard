About this Project & Agents

What is this project?

This dashboard is a meta project that documents all active projects and maps their relationships. It lives in the Claude Projects folder and is maintained with Claude Code — serving as a central hub for tracking ongoing work, dependencies, and synergies.

What can I use it for?

  • Ask cross-project questions: “Which projects use Utopia?”, “Where are there token conflicts?”
  • Build connections: Transfer insights from one design system to another
  • Track status: Last update, scope, and progress of all projects at a glance
  • Provide context: Claude has access to all source code and memory files in this project

Cockpit, Radar & Bridge

Three specialized helpers for this meta project. Each has two invocation methods — see Architecture below.

Architecture: Skills vs. Agents

Claude Code offers two mechanisms for reusable instructions: Skills and Agents. This project deliberately uses both — as a hybrid setup.

  Skills (/slash) Agents (claude --agent)
Invocation /cockpit, /radar, /bridge claude --agent cockpit
Autocomplete on / Yes No
Cross-invocation No — sequential by user only Yes — Cockpit spawns Radar & Bridge as subagents
Parallel execution No Yes — Radar + Bridge simultaneously
Tool restrictions No — full access Yes — tools: field in frontmatter
Files ~/.claude/skills/*/SKILL.md ~/.claude/agents/*.md

Why both?

  • Skills for daily use: type /radar, update data, done. Fast, direct, no setup.
  • Agents for orchestrated workflows: claude --agent cockpit starts a session where Cockpit autonomously decides if and when it needs Radar or Bridge — and spawns them as subagents.

Limitation: Subagents cannot spawn further subagents. Only the main agent (Cockpit) can delegate. That’s why Cockpit is the coordinator — Radar and Bridge are pure workers.

Typical workflows:

  • Quick: /radar → Numbers updated
  • Targeted: /bridge nova prism → Comparison
  • Orchestrated: claude --agent cockpit → “Update the dashboard and check if Prism has new patterns that Nova could use” → Cockpit spawns Radar + Bridge, processes results, rebuilds dashboard

Synergies & Collaboration

Utopia Fluid Type — Shared Methodology

All three design systems (Nova, Prism, Helix) are migrating to Utopia-based fluid typography with CSS clamp(). Insights from one project (e.g., token override strategy in Prism) can be directly transferred to the others.

Potential: Shared Utopia config generator, cross-system best practices, common viewport viewer component

Redleaf Ecosystem — Nova + Social Proof + Forge

Three projects working on the same design system: Nova Fluid Type modernizes the typography, Social Proof extends the component library, Forge enables live preview.

Potential: Forge as a universal preview tool, Social Proof with fluid type values, unified prototyping workflow

Design Bridge — Documenting All Learnings

Experiences from fluid type migrations, token architectures, and AI workflows can be prepared as learning resources on the platform.

Potential: Case studies, Token Workshop content, AI workflow documentation as learning cards

Viewport Viewer — Universal Preview Tool

Forge, Prism Utopia Viewer, and Helix all use their own viewport preview tools. A generic tool could serve all three design systems.

Potential: Generic fluid type viewer tool, open source, design-system-agnostic

Token Architecture — Primitive vs. Semantic

Prism and Helix share the same challenge: overriding semantic tokens for fluid type without breaking primitives and line heights.

Potential: Shared token override pattern, documented architecture pattern

Prototyping-Workflow — AI + Figma MCP

Nova Social Proof uses Claude Code + Figma MCP for research-to-prototype workflows. Experiences can be documented as a learning card on Design Bridge.

Potential: Design Bridge workflow card “From Research to Prototype with AI”

The Handoff
Idea

AI-powered design-to-dev workflow for component documentation. The classic workflow is reversed: documentation and edge cases come before building in Figma, not after. At the end, the loop closes: the finished Storybook component is synced back to Figma.

Workflow:
  1. Prompt Setup — MD doc template + a11y checklists + meta-prompt
  2. Doc Creation — Structured MD docs with AI (edge cases from the start: loading, error, keyboard nav)
  3. Figma Implementation — Build component, layer names = code identifiers
  4. Live Validation — Figma MCP + Design System MCP validate Figma against docs in real time
  5. Handoff — Devs receive: Figma file (dev-ready) + MD docs (Storybook-ready)
  6. Feedback Loop — Finished Storybook component synced back to Figma via Claude Code

Accessibility Checks (3 Pillars)

  • ›5 Quick Checks — Keyboard, Zoom 400%, Screen Reader, Contrast, States
  • ›ACCESSIBILITY.md Pattern — AI Agent Skill (Semantic HTML, ARIA, Anti-Patterns)
  • ›Project Standard — Template, severity matrix, CI/CD integration

Spectrum Color
Idea

OKLCH-based color palette generation for accessible design systems. 22 weights, 000–999, adapted for the OKLCH color space — perceptually uniform, CSS-native, no blue hue shift. Weight 500 guarantees WCAG 4.5:1 on both white and black.

Building Blocks:
  1. Base Palette — 22 weights in 5 categories (Highlights, 1/4 Tones, Mid-Tones, 3/4 Tones, Shadows), 10 semantic variants
  2. OKLCH Adaptation — Generic → semantic tokens, L/C/H as separate custom properties, light/dark mirroring
  3. Adobe Leonardo — Open source contrast engine: input = contrast ratio, output = matching color
  4. Figma Generator — OKLCH palette generation directly in Figma

Connections

  • ›Prism Fluid Type — Already has OKLCH implemented, learnings transferable via Bridge
  • ›Nova — Could receive an OKLCH palette (currently Hex/RGB)
  • ›Helix — Multi-brand system benefits from universally interchangeable palettes
